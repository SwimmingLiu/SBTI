# Multi-Quiz Platform Design Spec

**Date:** 2026-04-10

## Context

当前仓库已经完成 SBTI 单题库复刻，但用户需求已经升级为“题库入口站”：

- 首页不再只承载 SBTI，而是承载多个测试入口
- 当前项目中的 SBTI 需要保留，并作为其中一个测试
- 新增两个完整复刻目标：
  - `SDTI`：`https://fmnst.net`
  - `HERTI`：`https://herti.netlify.app`

被动取证结果：

- `SDTI` 是单文件页面，题库、维度、结果逻辑、隐藏结果和图片路径都内嵌在入口 HTML 中，无远端 API
- `HERTI` 也是单文件页面，20 题、16 个人格原型、马氏距离匹配逻辑均内嵌在入口 HTML 中，无远端 API
- `SDTI` 存在一个远端结果图资源 `feminist.WEBP`
- `HERTI` 结果页主要由文本、排版和关系卡组成，没有额外结果图片资源

## 用户目标

从 0 到 1 把当前项目改造成一个多题库站点：

1. 首页提供多个测试入口
2. 保留并可继续访问 SBTI
3. 完整复刻 SDTI
4. 完整复刻 HERTI
5. 通过 Playwright 真机浏览器方式提取每个目标站的所有可能结果、图标/图片、文案与结构证据
6. 按功能块拆分 PR，逐个测试、发 PR、请求 `@copilot` review、修复后合并

## 非目标

- 不引入后端服务、数据库或账号体系
- 不把测评逻辑改造成远端 API
- 不做“抽象到任意题库都可零配置接入”的过度平台化

## 约束与假设

- 必须保留当前 SBTI 的既有能力：静态导出、结果图、微信小程序兼容逻辑、Playwright E2E
- 每次代码修改后都要执行：`npx playwright test` → `npm run package:out` → commit → push
- `impeccable` skill 当前会话不可用，因此 UI 优化改为手工做统一视觉升级
- 当前工作区存在未提交的“小程序入口隐藏”改动，后续实现必须保留其行为，不得回滚

## 方案对比

### 方案 A：三个测试各写一套独立页面与状态

优点：

- 实现最直接
- 不需要额外抽象

缺点：

- 首页入口、元数据、测试脚本、研究目录会重复
- 后续新增题库时维护成本更高

### 方案 B：轻量题库注册表 + 每题库自有实现（推荐）

优点：

- 首页、导航、元数据、研究脚本入口统一
- 每个题库仍保留自己的题目结构、结果算法和 UI 风格，不会被过度抽象绑死
- 能在不破坏 SBTI 的前提下逐步接入 SDTI/HERTI

缺点：

- 需要先做一次壳层重构

### 方案 C：把三套题库强行抽象为同一套通用引擎

优点：

- 表面上最“平台化”

缺点：

- 三个站点的交互模型差异很大：
  - SBTI：完整列表答题 + 条件插题 + 图片结果
  - SDTI：整页答题 + 维度条 + 隐藏触发
  - HERTI：单题逐步切换 + 加载屏 + 文学人格卡
- 为统一而统一会显著增加复杂度和回归风险

## 选型结论

采用 **方案 B：轻量题库注册表 + 每题库自有实现**。

核心思路：

- 统一“站点入口层”和“研究/抓取层”
- 各题库各自持有数据、算法、UI 和结果提取逻辑
- 只共享真正通用的部分：首页入口卡片、路由组织、静态资源目录约定、Playwright 运行方式

## 架构设计

### 1. 路由与站点结构

- `/`：多测试题首页
- `/tests/sbti`：现有 SBTI
- `/tests/sdti`：新增 SDTI
- `/tests/herti`：新增 HERTI

首页展示题库卡片、简介、题量、结果数、风格标签和进入按钮。

### 2. 题库注册表

新增一个轻量注册表，负责：

- 首页卡片数据
- 站点内部导航
- 测试 slug 与页面标题映射
- 后续研究脚本按 slug 输出到对应目录

注册表只管理元信息，不管理答题算法。

### 3. 题库实现边界

每个题库独立维护以下内容：

- `data`：题目、结果原型、资源映射
- `engine`：得分/匹配/隐藏结果逻辑
- `app`：交互流程和页面状态
- `capture`：生成所有结果的场景与浏览器提取逻辑

这样可以保证：

- SBTI 不因 SDTI/HERTI 接入而被过度改造
- SDTI/HERTI 可以忠实复刻原站特征
- E2E 用例可以精确覆盖各自行为

### 4. 研究与证据目录

- `research/sbti/*`
- `research/sdti/*`
- `research/herti/*`

每个题库统一保留：

- `raw/index.html`
- `raw/headers.txt`
- `results/*.json`
- `results/*.png`
- `results/index.json`

### 5. 浏览器抓取策略

#### SDTI

- 先从源码提取题目、维度、结果分支条件
- 再通过本地脚本生成“每种结果至少一个可达答案集”
- 最后用 Playwright 打开远端站点，逐个提交并抓取：
  - 结果名称
  - 结果文案
  - 结果注释
  - 维度百分比
  - 结果图片和署名
  - 整页截图

#### HERTI

- 先提取 16 个人格原型和 20 题向量
- 用本地搜索生成每个原型至少一个可达答案集
- 用 Playwright 驱动远端站点逐题点击，抓取：
  - 主人格 / 镜像 / 反面
  - epigraph / persona / tags / soul
  - 整页截图

### 6. UI 策略

允许轻微优化，但保持原站辨识度：

- 首页做统一视觉升级，突出“题库站”而非单页测试
- SBTI 延续当前绿色体系
- SDTI 采用冷灰/黑白纸面感
- HERTI 保留米色、文学、长卷阅读气质

### 7. PR 划分

#### PR1：多题库壳层与 SBTI 路由迁移

- 新首页
- 题库注册表
- SBTI 从根页迁到 `/tests/sbti`
- 基础导航/E2E 迁移

#### PR2：SDTI 复刻

- SDTI 数据、引擎、页面
- 远端结果场景求解与 Playwright 抓取
- SDTI 本地 E2E

#### PR3：HERTI 复刻

- HERTI 数据、引擎、页面
- 16 结果场景求解与 Playwright 抓取
- HERTI 本地 E2E
- 首页最终文案和静态打包收尾

## 验收标准

- 首页能看到 SBTI / SDTI / HERTI 三个入口
- 三个测试都能独立完成完整答题到结果展示
- `research/sdti/results` 与 `research/herti/results` 具备“所有可达结果”的 JSON 与截图
- Playwright E2E 能真实跑通首页与三个测试的关键路径
- 每个功能块都完成：计划 → 实现 → 浏览器验证 → PR → `@copilot` review → 修复 → 合并
